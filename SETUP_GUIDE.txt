# STEP-BY-STEP GUIDE

This guide is for deploying the ELN logger, configuring users, and extending parsers.

## 1. Server-side eLabFTW deployment

1. Copy `docker/docker-compose.example.yml` and `docker/.env.example` into your deployment directory.
2. Rename `.env.example` to `.env`.
3. Replace all placeholder secrets in `.env`.
4. Generate an eLabFTW secret key with the container helper and paste it into `.env`.
5. Start the stack with `docker compose up -d`.
6. Run the eLabFTW database initialization if needed.
7. Open the ELN in the browser and create or activate users.

## 2. Cluster-side installation for a user

1. Copy the repo to the cluster, for example into `~/bmdeln`.
2. Run `bash setup.sh`.
3. Open `~/.bmdeln/config.env`.
4. Set:
   - `ELABFTW_URL`
   - `ELABFTW_TOKEN`
   - `ELABFTW_VERIFY_SSL`
   - `ELABFTW_TIMEOUT`
   - `REAL_SBATCH`
5. Run `source ~/.bashrc`.
6. Test with `python3 ~/bmdeln/api/elabftw_client.py`.

## 3. Daily usage

Use:

```bash
bmdsubmit input_file submit_script
```

instead of:

```bash
sbatch submit_script
```

What happens:
1. The wrapper detects the code from the Slurm script.
2. The input parser extracts metadata.
3. An eLabFTW entry is created.
4. A temporary copy of the Slurm script is generated.
5. The temporary script gets an auto-injected exit trap that calls `scripts/bmdeln_epilog.py`.
6. The job is submitted through the real `sbatch`.
7. When the job finishes, the ELN entry is updated with status, results, and attached/reference files.

## 4. What changed in this revision

- `sbatch` is no longer hardcoded.
- eLabFTW config is fully environment-driven.
- request timeouts are enforced in the API client.
- failed file uploads now raise errors instead of silently passing.
- epilog injection uses a shell exit trap instead of a simple appended command.
- Molpro is wired into the full submit → epilog flow.
- ORCA, Amber, and OpenMolcas now have first-pass input and output parsers.
- large-file path sections in the ELN body are inserted idempotently.
- secrets are removed from the Docker example and moved to `.env`.

## 5. Parser extension model

Every parser module under `parsers/` now exposes four important items:

- `JOB_LABELS`
- `RESULT_LABELS`
- `TAG_FIELDS`
- `FILE_FIELDS`

### 5.1 Add a new field to the parser

Add the field to the dictionary returned by the input or output parser.

Example:

```python
metadata['solvent'] = parsed_value
```

### 5.2 Show the field in the ELN body

Add one tuple to `JOB_LABELS` or `RESULT_LABELS` in that parser module.

Example:

```python
('Solvent', 'solvent')
```

The shared formatter in `parsers/common.py` will then render it automatically.

### 5.3 Turn a field into an ELN tag

Add the field name to `TAG_FIELDS`.

Example:

```python
TAG_FIELDS = ['job_type', 'functional', 'solvent']
```

### 5.4 Treat a parsed path as a referenced file

Add the field name to `FILE_FIELDS`.

Example:

```python
FILE_FIELDS = ['geometry_file', 'orbital_file', 'trajectory_file']
```

The epilog will then try to upload small files and list larger ones as cluster paths.

## 6. Parser-specific notes

### CP2K
- coordinate files are pulled from `COORD_FILE_NAME`
- standard CP2K log patterns are used for final energy and convergence

### Molpro
- input and output parsing are integrated end-to-end
- method energies and HOMO/LUMO style information are captured when present

### ORCA
- supports standard `!` keyword parsing, `xyzfile`, `%qmmm`, and `%neb`
- the output parser currently focuses on final energies, SCF convergence, and orbital energies

### Amber
- the input parser reads `&cntrl` parameters from `mdin`
- the output parser reads final thermo blocks, averages, wall time, and performance summary

### OpenMolcas
- the parser currently targets `&GATEWAY`, `&SEWARD`, and `&RASSCF` workflows
- root energies and the `Happy landing!` marker are captured from the output

## 7. Release hygiene

When you share a release, do not zip your full working directory. Use one of these instead:

```bash
git archive --format=zip HEAD -o auto-eln-logger-release.zip
```

or

```bash
tar --exclude-vcs --exclude='__pycache__' --exclude='*.tar.gz' --exclude='*.zip' -czf auto-eln-logger-release.tar.gz auto-eln-logger/
```

## 8. Recommended next validation

Before group-wide rollout, test one successful job for each supported code using the sample-style workflows you already collected:

- CP2K
- Molpro
- ORCA
- Amber
- OpenMolcas

Then check:
1. correct code detection
2. correct output file guess
3. correct status on completion
4. attached files versus cluster-path references
5. sensible ELN tags
